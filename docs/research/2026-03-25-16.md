## 行业动态

### 1. Binoculars 深度解析 — 最具可操作性的零样本方法

14:00 报告已列出 Binoculars 的基本原理（双模型 PPL 比值）。本次深挖实现细节：

**核心公式**：`score = log_perplexity(observer) / cross_perplexity(observer←performer)`

- **Observer LLM** 计算文本的 log-perplexity（文本对该模型有多"意外"）
- **Performer LLM** 生成 next-token 预测分布，Observer 再评估这些预测有多"意外"（即 cross-perplexity）
- 低分 = 高概率 AI 生成；高分 = 高概率人类书写

**模型对选择（关键实现细节）**：
- 论文明确指出 **observer 和 performer 应性能接近**（与 speculative decoding 相反，后者需要强弱配对）
- 推荐对：Falcon-7B + Falcon-7B-Instruct；LLaMA-2-7B + LLaMA-2-13B
- **GitHub 代码 bug**：论文仓库中 performer/observer 角色在代码中被调换（performer 实际执行了 observer 的工作），社区已发现但未修复
- 来源：[HuggingFace Blog](https://huggingface.co/blog/dmicz/binoculars-text-detection), [arXiv 2401.12070](https://arxiv.org/abs/2401.12070), [OpenReview](https://openreview.net/pdf/9db8f0545050123ec666c32082f9a786bbd45f55.pdf)

**性能数据（补充 14:00 报告未列出的细节）**：
- ChatGPT 文本：FPR=0.01% 时 TPR>90%（未训练于 ChatGPT 数据）
- ESL（非母语英语写作者）：准确率 99.67% — 这对当前项目极其重要，因为 Turnitin/GPTZero 的最大争议就是对 ESL 的假阳性
- 与 DetectGPT/Fast-DetectGPT 对比：Binoculars 无需 perturbation/sampling 步骤，直接前向推理即可

**与 Fast-DetectGPT 的互补性**：
- Fast-DetectGPT（ICLR 2024）使用 conditional probability curvature：机器文本 curvature ≈ 3，人类文本 curvature ≈ 0
- 比 DetectGPT 快 340x，检出 80% ChatGPT 内容，误报 1%
- **两者可并行作为独立信号**：Binoculars 测量"两个模型对文本的共识度"，Fast-DetectGPT 测量"条件概率曲面的曲率"——不同的数学视角
- 来源：[arXiv 2310.05130](https://arxiv.org/abs/2310.05130), [GitHub](https://github.com/baoguangsheng/fast-detect-gpt)

### 2. 对抗攻击最新研究（项目刚发现的 dialogue/code/emoji bypass 的学术背景）

项目在 15 轮对抗测试中发现对话体、emoji、代码混合可绕过所有信号。最新学术研究证实这不是孤立问题：

**Emoji Attack（ICML 2025 Poster）**：
- 通过在文本中插入 emoji 来操纵 tokenization，扭曲 Judge LLM 的 embedding space
- 机制：emoji 导致 subword tokenizer 产生非预期分割，打破模型对"有害/AI 生成"文本的检测模式
- 攻击成功率在部分系统上达到 100%
- 来源：[arXiv 2411.01077](https://arxiv.org/html/2411.01077v2), [ICML 2025](https://icml.cc/virtual/2025/poster/45356)

**Emoti-Attack: 零扰动对抗攻击（arXiv 2502.17392, 2025-02）**：
- 使用 emoji 序列实现零扰动（zero-perturbation）攻击——文本语义完全不变，仅插入 emoji 即可逃避检测
- 来源：[arXiv 2502.17392](https://arxiv.org/html/2502.17392v1)

**Code-Switching 多语混合攻击**：
- 在单条文本中混合多种语言（甚至低资源语言如斯洛文尼亚语、瓦伦西亚语）可直接穿透安全检测
- 与项目发现的"代码混合文本 confuses PPL"完全一致

**TempParaphraser（EMNLP 2025）**：
- 温度操控型改写：通过调节生成温度来改写文本以逃避检测
- 来源：[ACL Anthology](https://aclanthology.org/2025.emnlp-main.1607.pdf)

**Token Probability-Based Attack（arXiv 2501.18998, 2025-01）**：
- 使用 embedding 模型扰动 token 概率分布，针对 Fast-DetectGPT 等模型
- 在 10 秒内即可将机器文本误分类为人类文本
- 来源：[arXiv 2501.18998](https://arxiv.org/abs/2501.18998)

**Gradient-Based Evader（USENIX Security 2025）**：
- 梯度引导的文本修改，黑盒攻击也只需要查询检测器少量次数
- 频率惩罚和出现惩罚调整是最高效的攻击策略（将检测分降至 0.5 以下同时保持文本质量）
- 来源：[USENIX](https://www.usenix.org/system/files/usenixsecurity25-meng.pdf)

## 技术机会

### 1. Binoculars 第五路信号实现方案（最高优先级）

14:00 报告建议了 Binoculars 集成方向。本次给出具体实现方案：

**模型对选择（针对项目约束）**：
- 项目已有 qwen3:4b（PPL 信号用）。Binoculars 需要**性能接近的两个模型**
- **推荐方案 A**：qwen3:4b (observer) + qwen3:1.7b (performer)。同系列不同规模，tokenizer 兼容
- **推荐方案 B**：llama3.2:1b (observer) + llama3.2:3b (performer)。已有 1b，只需额外拉取 3b
- **不推荐**：qwen3:4b + llama3.2:1b（不同系列，tokenizer 不兼容，cross-perplexity 无意义）

**实现步骤**：
1. 在 `perplexity.py` 启动时加载第二个模型（同系列不同大小）
2. 对输入文本分别通过两个模型前向推理
3. 计算 `binoculars_score = log_ppl(observer) / cross_ppl(observer←performer)`
4. 将 binoculars_score 作为第五路信号加入 fused score
5. 阈值：论文使用 0.9015 作为分界线（低于=AI，高于=human），需用 dataset.jsonl 校准

**资源开销**：
- 额外内存：~2GB（1.7b 量化模型）或 ~4GB（3b 量化模型）
- 额外推理时间：约 +50%（可通过仅在其他信号"uncertain"时触发来优化）

**注意论文代码 bug**：GitHub 仓库中 performer/observer 角色实现反了，直接参考论文公式而非代码。

### 2. 条件触发式多信号架构（解决推理成本问题）

当前四路投票每次分析都跑全部四个信号。建议改为级联触发：

```
Stage 1: Stat 信号（0ms，纯计算）
  → 如果 stat 强信号（>85 或 <15）→ 直接返回
Stage 2: PPL 信号（~2s，单模型推理）
  → 如果 stat + PPL 强共识 → 直接返回
Stage 3: LR 信号（~0ms，用 Stage 2 的 PPL 特征）
  → 如果三路强共识 → 直接返回
Stage 4: DeBERTa + Binoculars（~3s，仅在不确定时触发）
```

预期效果：70%+ 请求在 Stage 2 即可返回，平均延迟降低 40-60%。

### 3. 对话体/emoji/代码混合的防御策略

项目发现的三大绕过方式在学术上都有对应研究。建议实施预处理防御层：

**对话体防御**：
- 预处理：检测对话格式标记（`"`, `—`, `:`开头的行），提取纯叙述内容重新评估
- 分段评估：对话中的每段独白分别评分，取加权平均

**Emoji 防御**：
- 预处理：strip 所有 emoji 后重新评估（参考 Emoji Attack 论文的防御建议）
- 双评分：保留 emoji 版本和去除 emoji 版本分别评分，差异过大则标记为疑似对抗攻击

**代码混合防御**：
- 语言检测：用 langdetect/fasttext 检测文本语言分布
- 分段处理：英文段落和代码块分别评估
- 标记：代码占比 >30% 时标记为"混合内容，检测可信度降低"

### 4. Conditional Probability Curvature 作为轻量补充

Fast-DetectGPT 的条件概率曲率可作为独立于 Binoculars 的信号：
- 机器文本 curvature ≈ 3，人类文本 curvature ≈ 0
- 只需 **一个** 评分模型 + 一个轻量采样模型（不需要两个同级模型）
- 比原版 DetectGPT 快 340x
- 开源代码：[github.com/baoguangsheng/fast-detect-gpt](https://github.com/baoguangsheng/fast-detect-gpt)

### P1: 预处理防御层（半天工作量）

在 `perplexity.py` 的 `/analyze` 路由入口处增加文本预处理：

```python
def preprocess_for_detection(text: str) -> dict:
    # 1. Strip emoji → re-score
    # 2. Detect dialogue format → extract narrative
    # 3. Detect code blocks → separate evaluation
    # 4. Language detection → flag mixed-language

        "cleaned_text": ...,
```

## 风险预警

### 1. 对抗攻击已产业化（USENIX Security 2025 确认）

USENIX Security 2025 的梯度引导逃逸器论文证明：**10 秒内即可将机器文本伪装为人类文本**。这不再是学术实验，而是成熟的攻击工具。frequency_penalty 和 presence_penalty 调整被确认为最低成本的有效攻击手段。

**对项目的影响**：仅靠 PPL 单一信号无法抵御参数调优攻击。这进一步验证了多信号投票架构的必要性——攻击者需要同时骗过 4-5 个正交信号，难度指数级上升。

### 2. Emoji/对话体/代码混合 bypass 是系统性缺陷

ICML 2025 Poster（Emoji Attack）和 arXiv 2502.17392（Emoti-Attack）证明 emoji 攻击可以达到 100% 逃逸率。项目发现的 bypass 不是个别 bug，而是 tokenizer 层面的系统性弱点——emoji 打破了 subword 分割的假设。

**紧急程度**：中。这些攻击在真实用户场景中出现概率较低（正常写作不会大量插入 emoji），但如果面对刻意对抗则完全无防御。

### 3. Binoculars 实现的陷阱

- **模型对必须同系列**：不同 tokenizer 的模型对无法计算有效的 cross-perplexity
- **GitHub 代码有 bug**：performer/observer 角色反了，必须以论文公式为准
- **阈值需要重新校准**：论文阈值 0.9015 基于 Falcon-7B 对，换模型后必须重新校准
- **内存翻倍风险**：加载第二个模型后，M4 的 16GB 统一内存可能紧张（qwen3:4b 4-bit ~3GB + 第二模型 ~2GB + DeBERTa ~3GB + FAISS ~2GB ≈ 10GB）

### 4. 140 行 if-else 融合逻辑的维护风险加剧

CLAUDE.md 第 107 条已标记"规则工程到天花板"。如果加入 Binoculars 作为第五路信号，if-else 逻辑预计膨胀到 200+ 行。建议在加入第五路信号时，同时重构为可配置的加权投票系统（权重可从小样本数据中学习），而非继续手写规则。

### 5. 学术市场萎缩持续加速

新增数据点：Vanderbilt 弃用原因明确指向"正式学术文体和 ESL 假阳性"。PMC 2026 论文系统性质疑了检测器在学术场景的可信度。如果 Binoculars 的 ESL 友好特性（99.67%）能被集成并验证，这反而是差异化机会——"唯一对 ESL 友好的检测器"是强卖点。
